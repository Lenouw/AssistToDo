# Leçons — AssistToDo

Format : `[date] | ce qui a mal tourné | règle pour l'éviter`

- 2026-06-12 | Le LLM mettait les events sans date sur "aujourd'hui", et appliquait une date dictée pour un item à TOUS les items de la phrase. Côté Swift, `route()` faisait aussi `dueDate ?? Date()`. | Ne JAMAIS inventer de date. Prompt : date/heure d'un item ne se propage pas aux autres ; calendar exige une date réellement dictée, sinon reminders/local avec dueDate=null. Swift : filet de sécurité, jamais de `Date()` par défaut pour un event.
- 2026-06-12 | Apple Notes : impossible de créer de vraies cases à cocher (checklist) via AppleScript/`body` HTML. La seule voie (simulation clavier System Events) OBLIGE Notes au premier plan → vole le focus → rejeté par l'utilisateur ("préhistorique"). | Ne pas partir sur la frappe clavier pour écrire dans une autre app. Pour les courses : texte simple dans Apple Notes (un `<div>` par ligne, focus préservé). Cases cochables = seulement faisable dans une app qu'on contrôle. Présenter le triangle d'impossibilité (cases + Notes partagé + sans focus volé : choisir 2) AVANT de coder.
- 2026-06-12 | App Sandbox : accès Rappels échouait silencieusement (fallback local) car l'entitlement `com.apple.security.personal-information.reminders` manquait (seul `.calendars` était là). | EventKit Rappels ET Calendrier exigent CHACUN leur entitlement `personal-information.*`. Vérifier les deux quand on ajoute une intégration Apple sandboxée.
- 2026-06-12 | NotesService.escape() n'échappait pas le backslash ni les sauts de ligne → un article avec `\` ou `\n` cassait/injectait l'AppleScript. | Pour une string injectée dans AppleScript + HTML : échapper backslash D'ABORD, puis guillemet, puis HTML, puis neutraliser `\n\r`. Tester avec entrées hostiles.
- 2026-06-12 | TaskParser défautait `dueDate` à aujourd'hui pour TOUTES les destinations → neutralisait le garde-fou anti-date-inventée et posait des events all-day sur aujourd'hui. | Ne défauter à "aujourd'hui" que pour les tâches locales ; calendar/reminders/notes gardent dueDate=nil si rien n'est dicté.
- 2026-06-12 | Note de courses des Réglages ignorée : le LLM renvoyait un `noteName` deviné qui écrasait la note configurée (→ écriture/création dans une mauvaise note). | Rendre le réglage utilisateur autoritaire ; ignorer le `noteName` du LLM pour les courses (sauf besoin explicite vérifié).
- 2026-06-12 | Articles de courses notés "Acheter X" au lieu du nom seul. | Prompt notes : `text` = nom de l'article seul, jamais de verbe.

- 2026-06-13 | Extraction couche partagée macOS→iOS : le projet Xcode 16 utilise un PBXFileSystemSynchronizedRootGroup (groupe synchronisé) → 0 référence fichier individuelle dans project.pbxproj. Déplacer un fichier hors du dossier suffit à le retirer du target (pas de chirurgie pbxproj pour retirer). Pour ajouter une couche partagée : `git mv` les fichiers vers un nouveau package SwiftPM local + `import <Module>` dans les consommateurs. Vérifié : build macOS reste vert.
- 2026-06-13 | Package SwiftPM local dépendant de WhisperKit en plus du projet Xcode : conflit "whisperkit is required using two different revision-based requirements" même avec le MÊME SHA, car le projet épingle en `branch main` et le package en `revision:`. Règle : un package partagé doit déclarer la dépendance commune avec le MÊME TYPE de contrainte que le projet (ici `branch: "main"`), le SHA exact restant verrouillé par Package.resolved du projet.
- 2026-06-13 | `.iOS(.v18)` dans Package.swift exige swift-tools-version 6.0 (qui bascule en mode langage Swift 6 strict, risqué). Le minimum d'un *package* peut rester inférieur (.v17) en tools 5.10 ; la cible *app* iOS 18 se règle dans Xcode (IPHONEOS_DEPLOYMENT_TARGET), indépendamment du minimum du package.
- 2026-06-13 | Créer des targets Xcode en CLI : gem `xcodeproj` (gem install --user-install), lire le pbxproj avec `ruby -E UTF-8:UTF-8` (sinon "invalid byte sequence in US-ASCII" sur les commentaires accentués). new_target(:application/:app_extension), add_file_references aux deux targets pour le code partagé, copy_files_build_phase(:plug_ins) pour embarquer l'extension, XCScheme.save_as pour un scheme buildable. Info.plist via INFOPLIST_KEY_* + GENERATE_INFOPLIST_FILE (pas de plist manuel).

- 2026-06-13 | Audit sécu iOS : (1) le token Bearer Toudou partait vers une URL base réglable par l'utilisateur SANS validation de schéma → un http:// exfiltrait le token en clair. Règle : un secret (Bearer/clé) ne part QUE vers https:// validé ; ignorer/replier toute URL non-https. (2) PreferencesService.export() écrivait la clé OpenRouter en CLAIR dans le JSON exporté → fuite hors Trousseau. Règle : ne jamais round-tripper un secret dans un fichier exporté en clair (exclure du payload). (3) Keychain en AfterFirstUnlock → passé en ...ThisDeviceOnly (hors sauvegardes/Trousseau iCloud) pour des secrets device-local.
- 2026-06-13 | Audit fonctionnel capture iOS : (1) reset de champs protégés par NSLock fait HORS lock dans AudioCapture.start() = data race avec le tap audio. (2) begin() enregistrait même WhisperKit pas prêt (1er run = téléchargement modèle) → capture perdue à la transcription : bloquer si !isReady. (3) demander la permission micro pendant le geste press-and-hold (await) = course si l'utilisateur relâche avant begin() ; demander la permission À L'APPARITION de la vue, garder le geste 100% synchrone.

- 2026-06-20 | Transcription WhisperKit iPhone, plusieurs pièges enchaînés : (1) `prewarmModels()` lancé en `Task` bg APRÈS `isReady=true` → une capture pendant le préchauffage faisait tourner transcribe() + prewarm en parallèle sur la même instance (non réentrante) = blocage indéfini sans timeout. Fix final : SUPPRIMER prewarm (WhisperKit(config) charge/compile déjà via loadModels ; prewarm n'est qu'un bonus risqué qui se fige sur certains modèles/ANE). isReady dès le chargement, compilation résiduelle à la 1ʳᵉ transcription (bornée par timeout 90 s). (2) Bandeau de préparation SANS pourcentage = l'utilisateur croit à un bug (attente opaque ~480 Mo). Exposer une vraie progression via `WhisperKit.download(variant:progressCallback:)` puis init avec `download:false, modelFolder:`. (3) Modèles large (distil-large-v3-turbo) inadaptés iPhone : compile ANE = minutes/blocage. Défaut iPhone = "small". (4) print() invisible sur device : utiliser os.Logger (subsystem com.assisttodo) → lisible dans Console.app device branché.
- 2026-06-20 | Routage calendrier/rappels iOS = SILENCIEUSEMENT dépendant de la clé OpenRouter. Sans clé (ou erreur HTTP/timeout), TaskParser renvoie un fallback texte brut → destination .local → la capture atterrit dans "À faire", jamais en calendrier. Symptôme trompeur : "transcription marche mais aucun event créé, tout est bugué". Règle : quand une feature dépend d'un secret saisi par l'utilisateur, logguer EXPLICITEMENT la raison du fallback (noKey / httpStatus) au lieu d'échouer en silence, et vérifier ce maillon en premier dans le diagnostic. Clé entrée in-app (Réglages → Keychain), jamais bakée dans le build partagé.

- 2026-07-02 | INCIDENT calendrier fantôme : un rejeu automatique de captures (« aucune idée perdue ») sans borne temporelle a rejoué de VIEUX audios de test (12 jours) avec now=aujourd'hui → création répétée d'événements calendrier fantômes à chaque foreground. Règles : (1) tout rejeu/retry AUTOMATIQUE d'actions à effets externes (calendrier, rappels, réseau) DOIT être borné (fenêtre temporelle + compteur de tentatives) ; (2) un rejeu doit résoudre les dates relatives au moment de la DICTÉE (now = createdAt de la capture), jamais au moment du rejeu ; (3) avant d'élargir un mécanisme de replay, inventorier ce que le journal contient DÉJÀ chez l'utilisateur (données historiques = munitions).
- 2026-07-02 | Un fallback silencieux (CaptureStore → in-memory via `try? … ?? inMemory`) fait « disparaître » les données aux yeux de l'utilisateur sans aucun signal. Règle : tout repli dégradé qui perd de la persistance doit être VISIBLE dans l'UI (bandeau) + loggé en .fault.

- 2026-07-02 | Bug capture perdue = @Published muté pendant le cycle de vie de la vue. `prepare()` faisait `model.autoStartCapture = false` ; comme SwiftUI relit ce @Published pour construire la sheet, la vue basculait de "auto-start" (bouton Terminer→end()) à "manuel" (cercle→cancel()) EN PLEIN enregistrement → cancel() écrit l'audio mais ne journalise pas. Règle : ne JAMAIS muter un état observé qui pilote la config d'une vue active pendant que cette vue est à l'écran ; figer la valeur d'entrée d'une sheet par identité (`@State frozen = param` en onAppear) au lieu de relire le modèle. Corollaire : deux chemins de fin (end/cancel) où l'un journalise et l'autre non = piège ; un chemin d'annulation doit rester atteignable UNIQUEMENT par une action explicite d'annulation.
- 2026-07-02 | Méthode de diagnostic device qui a marché : `xcrun devicectl device install/process launch --console` pour installer+lancer sur l'iPhone réel, `xcrun devicectl device copy from --domain-type appDataContainer` pour RAPATRIER le vrai store SwiftData, puis `sqlite3` dessus (tables Z*, compteurs Z_PRIMARYKEY) pour la vérité terrain. Piège : `print()` vers un stdout non-TTY est bufferisé → invisible ; soit `setvbuf(stdout,nil,_IONBF,0)`, soit reproduire l'échec en LOCAL avec le store rapatrié (un test XCTest a prouvé que le save() marchait → le bug était en amont, pas dans la persistance).
